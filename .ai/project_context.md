# 프로젝트 기반 지식 (Vibe Coding Context)

안녕하세요! 이 프로젝트는 **"바이브 코딩(Vibe Coding)"** 방식으로 개발되고 있는 파이썬 디스코드 봇(Discord.py)입니다.
메인 개발자인 유저(User)는 아이디어 기획과 프롬프팅, 검수를 주로 수행하며, 시스템의 실제 코드 작성과 아키텍처 구성은 주로 AI(당신)가 담당합니다.

따라서 미래의 AI 파트너가 이 프로젝트에 기여할 때 반드시 지켜야 할 일관성 원칙을 아래에 명시합니다.

---

## 🏗️ 1. 아키텍처 및 책임 분리 (Architecture & SRP)
우리의 봇은 단일 책임 원칙(SRP)을 철저하게 준수하고 있습니다. 코드를 추가하거나 수정할 때 항상 이 구조를 따르세요.

1. **`main_bot.py`**: 봇의 진입점. Cog 로딩, 이벤트 등록, 봇 인증만 담당합니다. (실제 비즈니스 로직을 여기에 넣지 마세요!)
2. **`database_manager.py`**: SQLite 연동 및 쿼리의 중심입니다. 데이터베이스 락(Lock)을 방지하고 연결을 일원화하기 위해 DB 코드는 모두 여기에 둡니다. Cog 내에서 커넥션을 직접 만들지 않습니다.
3. **`cogs/` (기능 모듈 폴더)**: 새로운 기능은 반드시 Cog 형태로 분리합니다.
    - Music (`cogs/music/`): 음악 로직은 컨트롤러(`music_agent`), 코어재생기(`music_core`), 인터페이스(`music_ui`)로 쪼개져 있습니다.
    - Summary (`cogs/summary/`): Gemini AI 요약 기능. 리스너 엔진과 AI 에이전트 통신이 나뉘어 있습니다.
    - Leveling (`cogs/leveling/`): 경험치, 레벨, 랭킹 및 역할 부여를 통제합니다.
    - Logging (`cogs/logging/`): 디스코드 원격 알림을 포함한 무중단 로깅(File+Console)을 전역으로 책임집니다.
4. **`scripts/` 및 `docs/`**: 백업/업데이트 쉘 스크립트(.sh)나 마크다운 문서는 코어 코드와 섞이지 않도록 격리합니다.

## 🧑‍💻 2. 바이브 코딩 (Vibe Coding) 가이드라인
*   **제안과 프로토타입**: 사용자가 추상적인 아이디어를 내면, 단순히 묻기만 하지 말고 가능한 아키텍처를 스케치하여 제안(Prompt)하세요.
*   **불필요한 수정 방지**: 요구된 코드 이외의 멀쩡한 코드를 "개선" 명목 하에 마음대로 리팩토링하지 마세요.
*   **에러 핸들링**: 새로운 기능을 만들 땐 항상 로거(`logger.error`)를 사용하고 `try/except`로 봇이 멈추는 것을 방지하세요.

## 📝 3. 깃허브 커밋(Git Commit) 메시지 규약
일관된 커밋 히스토리를 위해 **Angular Commit Convention**을 사용합니다. AI가 커밋을 생성할 때는 반드시 아래 포맷을 따르세요.

### 포맷:
```text
<타입>[옵션 필드]: <설명>

<본문 (선택 사항)>
```

### 타입 정의:
*   `feat`: 새로운 기능 추가 (예: `feat: 레벨링 Cog 추가`)
*   `fix`: 버그 수정 (예: `fix: 음악 스킵 오류 해결`)
*   `refactor`: 코드 리팩토링 (기능 변화 없음) (예: `refactor: music_utils 로직 분리`)
*   `docs`: 문서 수정 (README.md 등) (예: `docs: API 명세서 업데이트`)
*   `style`: 코드 포맷팅, 세미콜론 누락, 들여쓰기 수정 (코드 동작 변경 없음)
*   `chore`: 빌드 업무 수정, 스크립트 이동, 패키지 매니저 세팅 등 (예: `chore: auto_backup 스크립트 이동`)
